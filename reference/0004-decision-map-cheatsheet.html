Decision Map 速查表

decision-mapping 把一个模糊想法拆成一张紧凑的问题地图。地图不是最终方案，而是探索未知数的路线。

三种票据

类型什么时候用常搭配的 skill产物
Discuss 问题主要靠和用户追问、澄清取舍来解决。 grilling, domain-modeling 票据里的 Answer，可能附带术语或决策记录。
Research 问题依赖外部知识、第三方 API 文档、已有资料或本地知识库。 资料检索、代码阅读、文档阅读 一份 markdown 研究摘要，链接回 decision map。
Prototype 问题必须通过试出来回答，例如交互感觉、状态机行为、UI 方案。 prototype 一次性原型，以及从原型里得到的结论。

票据格式

## #1: 选择邀请过期策略

Blocked by: #<ticket-number>
Type: Research | Prototype | Discuss

### Question

邀请链接应该何时过期，过期后用户看到什么？

### Answer

<解决后再填写>

和 grilling 的搭配

动作谁负责结果写到哪里
把松散想法拆成多个未知数 decision-mapping 新的 decision map 文件
追问某个 Discuss 票据 grilling 该票据的 Answer
发现新的依赖问题 decision-mapping 新增票据，并填写 Blocked by
确认不需要继续建图 decision-mapping 转入 to-prd、to-issues 或实现阶段
判断规则：如果一次讨论就能看清路线，不需要 decision map。只有当模糊想法需要跨多次 agent 会话、研究或原型来推进时，才值得建图。
