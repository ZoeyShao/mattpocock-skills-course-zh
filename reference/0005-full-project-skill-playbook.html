完整项目 Skill Playbook

这张表把后续课程压缩成一个可执行路线：每个阶段先看产物，再选 skill。

阶段什么时候进入主要 skills完成信号
发现追问想法松散，有未知数。decision-mapping, grilling, prototype, handoff关键未知数被回答，或被拆成 Research/Prototype/Discuss 票据。
需求与领域语言方向清楚，需要写成可验收需求。to-prd, domain-modeling, grill-with-docsPRD、领域术语、必要 ADR 就位。
架构与接口要决定模块形状、测试 seam 或重构方向。codebase-design, improve-codebase-architecture接口判断、deepening 候选或架构方向清楚。
切片与排期PRD 或计划要变成可执行任务。to-issues, triage, scaffold-exercises每个 issue 都是可验证的纵向切片，状态标签正确。
实现与诊断开始写代码、修 bug 或迁移测试。implement, tdd, diagnosing-bugs, migrate-to-shoehorn代码完成、测试通过、bug 有回归保护。
质量门禁准备合并、提交或保护 repo。review, setup-pre-commit, git-guardrails-claude-code, resolving-merge-conflictsdiff 过 review，自动检查或安全钩子到位。
沉淀交接上下文太长、要换 agent、要写文章或沉淀流程。handoff, obsidian-vault, teach, writing-great-skills, writing-*下一位 agent 或未来的你能接住，不需要重新考古。
总规则：不要问“我想用哪个 skill”。先问“当前阶段的下一份耐久产物是什么”。产物决定 skill。
