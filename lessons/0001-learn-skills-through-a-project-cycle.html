项目周期总览

第 0001 课 · 工程范式 · 12 分钟

这套课程不按 skill 名字排列，而按真实项目周期组织。一个 skill 的意义，来自它在项目阶段里解决的问题。

项目周期视角

把一次工程工作粗略看成八个阶段：

  1. 工作区准备
  2. 发现与澄清
  3. 需求与领域语言
  4. 架构与接口设计
  5. 切片与排期
  6. 实现与验证
  7. 质量门禁
  8. 知识沉淀与交接

参考：项目周期 Skill 地图。

关键判断

面对一个请求，不要先问“我记得哪个 skill”。先问三个问题：

  1. 现在处于哪个项目阶段？例如发现、需求、设计、实现、review。
  2. 这个阶段的产物是什么？例如 PRD、issue、测试、review 结论、handoff。
  3. 这个 skill 是否能直接推动产物成形？如果不能，它可能只是看起来相关。

例子：一个功能从想法到完成

“我想给应用加一个团队邀请功能。”

早期你可能用 decision-mapping 澄清未知数，用 to-prd 写需求，用 domain-modeling 固定术语。进入设计后，可能用 codebase-design 判断接口和 seam。拆工作时用 to-issues。实现时用 tdd 或 implement。遇到 bug 用 diagnosing-bugs。完成后用 review 检查，再用 handoff 交接上下文。

回忆练习

先判断阶段，再选 skill。想好后揭晓答案。

1.“我们知道要做邀请功能，但还不知道邀请过期、权限和邮件失败怎么处理。”

阶段：发现与澄清。可能用 decision-mapping 拆未知数，或用 grilling 追问边界。

2.“需求差不多清楚了，请把它变成可验收的文档。”

阶段：需求与领域语言。可能用 to-prd，如果术语混乱则搭配 domain-modeling 或 grill-with-docs。

3.“PRD 已经有了，现在要拆成几个 agent 可以独立完成的任务。”

阶段：切片与排期。可能用 to-issues，目标是得到独立可抓取、可验收的小任务。

主要来源

本课基于 mattpocock/skills 仓库中非 deprecated skills 的 frontmatter description 和工作流说明，并压缩成 项目周期 Skill 地图。后续每进入一个阶段，我们会打开该阶段最重要的渲染版 SKILL.md 细读。
