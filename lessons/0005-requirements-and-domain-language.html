需求与领域语言

第 0005 课 · 工程周期第 2 阶段 · 15 分钟

当关键未知数已经足够清楚，下一步是把它们变成可验收、可交给 agent 或人类实现的需求。

to-prd 的位置

to-prd 把当前上下文合成为 PRD，并发布到 issue tracker。它不重新采访用户，而是整理已经知道的内容：问题、方案、用户故事、实现决策、测试决策、范围外事项。

如果 issue tracker 或 triage 标签还没配置，应先回到第 0 阶段。

领域语言为什么跟 PRD 绑定

PRD 不是只给人读，也给后续 agent 读。术语不稳，后续 issue、测试名、接口名都会漂移。这里会用到：

这里要区分两个文档角色：领域语言负责固定“这些概念叫什么、边界在哪里”；ADR 负责记录“我们为什么选择这种设计”。如果 PRD 里出现重要技术取舍，就可能需要补 ADR。

  • domain-modeling：当场挑战模糊词，更新 CONTEXT.md，必要时写 ADR。
  • grill-with-docs：一边追问，一边对照现有领域文档和 ADR。

domain-modeling 和 grill-with-docs 怎么选

Skill 它主要解决什么 什么时候用
domain-modeling 把领域概念变清楚：统一术语、划清边界、记录少量重要 ADR。 当问题集中在“这个东西到底叫什么、和另一个概念有什么区别、这个决策该不该记下来”。
grill-with-docs 把一个具体方案追问清楚，并用已有 CONTEXT.md 和 ADR 检查它是否自洽。 当你已经有计划或设计，但担心边界、例外、术语和历史决策没有被问透。

简化判断：词和模型乱，用 domain-modeling；方案需要被拷问，用 grill-with-docs。两者都需要时，先用 domain-modeling 定词，再用 grill-with-docs 压方案。

判断顺序

  1. 术语是否清楚？不清楚先 domain-modeling。
  2. 需求是否可验收？不清楚继续 grill-with-docs。
  3. 上下文是否足够？足够后用 to-prd。

小测

“用户、成员、账号在对话里混着用。”下一步是什么？

先用 domain-modeling 固定术语；如果还要边追问边更新文档，用 grill-with-docs。
