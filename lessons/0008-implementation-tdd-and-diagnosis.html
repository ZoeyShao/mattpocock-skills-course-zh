实现、TDD 与诊断

第 0008 课 · 工程周期第 5 阶段 · 15 分钟

实现阶段最容易变成“读代码、猜原因、直接改”。这组 skills 的共同目的，是建立短反馈循环。

implement 是总入口

implement 接收 PRD 或 issue，执行实现。它会尽可能使用 tdd，定期跑 typecheck、单测，最后跑完整测试，并用 review 回看工作。

tdd：一个行为一个循环

tdd 不是“一次写完所有测试”。它是红灯、绿灯、重构：一个行为，一个失败测试，最少实现，通过后再下一个行为。测试应该验证公开行为，而不是内部细节。

diagnosing-bugs：先造反馈循环

diagnosing-bugs 面向困难 bug。第一阶段不是猜原因，而是建立一个能稳定复现用户症状的 pass/fail loop。没有 loop，就不要进入假设。

migrate-to-shoehorn

这是一个窄技能：把测试里的 TypeScript as 断言迁移到 @total-typescript/shoehorn。只用于测试代码。

小测

“线上偶发慢，但本地看代码猜不出。”先做什么？

用 diagnosing-bugs 建立可运行的反馈循环或性能基线。先量，再猜。
