Repo 准备契约

setup-matt-pocock-skills 的作用，是把一个 repo 里“agent 需要知道但不能瞎猜”的约定写清楚。

契约回答什么问题写到哪里哪些 skills 会依赖
Issue tracker 任务、PRD、QA 反馈应该发布到哪里？GitHub、GitLab、本地 markdown，还是其他系统？ docs/agents/issue-tracker.md to-prd, to-issues, triage
Triage labels 五个标准 triage 状态，在这个 repo 的 issue tracker 里实际叫什么标签？ docs/agents/triage-labels.md triage, to-issues
Domain docs 领域语言和架构决策在哪里？这是单上下文 repo，还是多上下文 repo？ docs/agents/domain.md diagnosing-bugs, tdd, improve-codebase-architecture
Agent skills block agent 进入 repo 后，应该从哪里找到这些约定？ CLAUDE.md 或 AGENTS.md 所有需要 repo 约定的工程类 skills
判断规则：如果一个 skill 会读写 issue、使用 triage 标签、依赖领域词汇或 ADR，那么在新 repo 中第一次使用它之前，先考虑是否需要跑 setup-matt-pocock-skills。

小词典

术语意思常见结构
单上下文 repo 整个仓库基本共享一套领域语言和架构决策。agent 读根目录的 CONTEXT.md 和 docs/adr/ 就够了。 /CONTEXT.md
/docs/adr/
多上下文 repo 一个仓库里有多个相对独立的业务或技术上下文，例如 frontend、billing、ordering。每个上下文可能有自己的领域词汇和 ADR。 /CONTEXT-MAP.md
/src/billing/CONTEXT.md
/src/ordering/CONTEXT.md
上下文 一组词汇、规则和设计决策共同适用的范围。它回答：“这些概念在哪个边界内成立？” 一个 app、一个 bounded context、一个 package，或 monorepo 里的一个子系统。
ADR Architectural Decision Record，架构决策记录。它记录一个重要技术决定的背景、决定内容、理由和后果。领域语言回答“叫什么”，ADR 回答“为什么这样设计”。 /docs/adr/0001-use-postgres.md
/src/billing/docs/adr/0002-money-model.md
适合记录有明显取舍的决定，不适合记录随手可见的小实现细节。
