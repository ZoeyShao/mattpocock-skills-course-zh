项目周期 Skill 地图

把 skills 当成工程周期里的工具，而不是孤立命令。先判断项目处在哪个阶段，再选择合适的 skill 组合。

阶段核心问题典型产物可能用到的 skills
0. 工作区准备 这个 repo 是否已经具备让 skills 正常工作的上下文？ Agent 配置、issue 追踪约定、领域文档位置 setup-matt-pocock-skills, ask-matt
1. 发现与澄清 我们到底要解决什么问题？有什么未知数？ 决策地图、追问记录、原型、问题边界 decision-mapping, grilling, grill-me, prototype
2. 需求与领域语言 需求如何验收？项目里这些词是什么意思？ PRD、领域术语、上下文文档、必要 ADR to-prd, domain-modeling, grill-with-docs
3. 架构与接口设计 模块边界、公共接口和测试方式应该长什么样？ 接口判断、架构决策、deep module 机会 codebase-design, improve-codebase-architecture
4. 切片与排期 怎样把工作拆成可以独立完成、独立验证的小块？ issues、triage 状态、反馈项、练习目录 to-issues, triage, scaffold-exercises
5. 实现与验证 如何用最短反馈循环完成代码，并证明行为正确？ 代码改动、测试、bug 复现与修复、实现记录 tdd, implement, diagnosing-bugs, migrate-to-shoehorn
6. 质量门禁 代码是否符合规范、需求和团队安全约束？ review 结论、pre-commit 配置、冲突解决、风险清单 review, setup-pre-commit, git-guardrails-claude-code, resolving-merge-conflicts
7. 知识沉淀与交接 如何让未来的自己或下一个 agent 接得住？ handoff、文章、笔记、课程、改进后的 skills handoff, obsidian-vault, edit-article, writing-shape, teach, writing-great-skills
使用规则：真实项目通常会来回跳阶段。比如实现时发现概念混乱，就回到“需求与领域语言”；review 时发现设计不稳，就回到“架构与接口设计”。
