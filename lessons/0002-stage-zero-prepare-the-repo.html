第 0 阶段：准备 Repo

第 0002 课 · 工程周期第 0 阶段 · 12 分钟

在真实项目里，很多 skills 不是缺能力，而是缺上下文。它们不知道 issue 发到哪里，不知道 triage 标签叫什么，也不知道领域语言和架构决策在哪里。setup-matt-pocock-skills 就是用来补齐这些 repo 级约定的。

这个 skill 解决什么问题

setup-matt-pocock-skills 会为当前 repo 搭出一份“agent 使用契约”。它通常会写四类东西：

  • CLAUDE.md 或 AGENTS.md 里的 Agent skills 区块。
  • docs/agents/issue-tracker.md：issue 和 PRD 应该去哪里。
  • docs/agents/triage-labels.md：五个标准 triage 状态对应哪些真实标签。
  • docs/agents/domain.md：领域文档和 ADR 应该从哪里读。

参考：Repo 准备契约。

为什么它排在第 0 阶段

因为它不是项目中的一个业务步骤，而是让后面的步骤少猜。比如：

  • to-prd 和 to-issues 要知道“发布到 issue tracker”到底意味着什么。
  • triage 要知道“ready-for-agent”在你的 tracker 里叫什么标签。
  • diagnosing-bugs、tdd、improve-codebase-architecture 要知道领域词汇和 ADR 在哪里。

术语：ADR 是什么

ADR 是 Architectural Decision Record，中文可以叫“架构决策记录”。它是一篇短文，用来保存团队做过的一个重要技术或架构决定。

一个 ADR 通常回答四件事：当时的背景是什么，决定了什么，为什么这么决定，以及这个决定带来的后果。它不是“最终真理”，而是给未来的人和 agent 留下当时的判断依据。

适合写成 ADR 的问题通常带有取舍：为什么选 PostgreSQL 而不是 MongoDB，为什么订单模块使用事件溯源，为什么前端状态放在 URL 里。不适合写 ADR 的，是随手能从代码看出来的小实现细节。

领域语言回答“这些东西叫什么、边界在哪里”；ADR 回答“为什么我们决定这样设计”。agent 读 ADR 的目的，是避免无意中推翻过去的决定。如果新方案确实要推翻旧决定，也应该明确说出来，而不是悄悄绕开。

它的工作流

  1. 探索：查看 git remote、AGENTS.md/CLAUDE.md、CONTEXT.md、docs/adr/、docs/agents/ 等。
  2. 逐项询问：一次只问一个决定：issue tracker、triage 标签、domain docs 布局。
  3. 确认草稿：先展示将要写入的区块和三个文档。
  4. 写入：更新已有的 CLAUDE.md 或 AGENTS.md，并写入 docs/agents/*.md。

不要误用它

它不是每次开工都要跑。它适合在新 repo 第一次使用工程类 skills 前运行，或者当这些 skills 明显不知道 issue tracker、标签或领域文档位置时运行。

回忆练习

判断是否需要先用 setup-matt-pocock-skills。

1.“这个 repo 没有 AGENTS.md，我们要用 to-issues 把 PRD 发成 GitHub issues。”

需要。to-issues 要知道 issue tracker 是 GitHub，并且需要 repo 里的 agent 约定入口。

2.“repo 已经有 docs/agents/issue-tracker.md 和 triage-labels.md，现在只是修一个小 bug。”

通常不需要。直接进入 diagnosing-bugs 或普通修复流程即可；除非现有约定明显过期。

3.“团队把 issues 从 GitHub 迁到了 Linear，但 docs/agents 还是旧内容。”

需要。issue tracker 约定变了，应重新更新 docs/agents/issue-tracker.md，否则后续 skills 会写错地方。

主要来源

本课基于 setup-matt-pocock-skills 的 SKILL.md，以及它附带的 issue tracker、triage labels 和 domain docs 模板。该 skill 属于上游非 deprecated 清单。
