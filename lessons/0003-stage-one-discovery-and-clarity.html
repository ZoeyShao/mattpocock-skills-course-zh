第 1 阶段：发现与澄清

第 0003 课 · 工程周期第 1 阶段 · 15 分钟

第 1 阶段的任务，不是立刻写 PRD，也不是急着拆 issue。它的任务是把“我们大概想做什么”变成“我们知道有哪些未知数，以及应该按什么顺序解决它们”。

主 skill：decision-mapping

decision-mapping 适合处理需要多次 agent 会话才能变清楚的松散想法。它会产出一份 compact markdown 文件，里面是一组按依赖排序的问题票据。

这份地图是探索路线，不是最终计划。它故意允许远处还在“战争迷雾”里；我们只需要推进当前边界上的问题。

参考：Decision Map 速查表。

什么时候用它

如果用户说的是“我有个想法，但还不知道怎么落地”，先判断模糊程度：

  • 如果几轮追问就能清楚，直接讨论，然后进入 to-prd 或实现。
  • 如果需要研究、原型、多人或多次会话，使用 decision-mapping。
  • 如果问题主要是视觉、交互或状态行为的手感，后续票据可能调用 prototype。

三种问题票据

  • Discuss：靠追问和取舍解决。常搭配 grilling 或 domain-modeling。
  • Research：靠读资料解决，比如第三方 API 文档、竞品、规范、本地知识库。
  • Prototype：靠试出来解决，比如 UI 方案、状态机、交互流程。

好的票据应该小到一个 agent 会话能解决。太大的问题要继续拆。

decision-mapping 和 grilling 怎么搭配

decision-mapping 管整张地图，grilling 管一张 Discuss 票据里的深挖。

常见节奏是：

  1. 先用 decision-mapping 把松散想法拆成一组票据。
  2. 遇到 Type: Discuss 的票据时，用 grilling 一次只问一个问题，沿着设计树往下追。
  3. 当这张票据的问题被回答，把结论写回 decision map 的 Answer。
  4. 如果追问过程中发现新未知数，就新增票据，并写清楚 Blocked by 依赖。

一句话：decision-mapping 决定“接下来要问哪些问题、按什么顺序问”，grilling 决定“这个问题要怎么问到足够清楚”。

一个例子

松散想法：“我想做一个团队邀请系统。”

可能的 decision map：

## #1: 邀请的核心对象是什么？
Blocked by:
Type: Discuss

## #2: 邀请链接应该如何过期？
Blocked by: #1
Type: Discuss

## #3: 邮件服务选哪家？
Blocked by: #1
Type: Research

## #4: 邀请流程的 UI 手感如何？
Blocked by: #1, #2
Type: Prototype

和后续阶段的关系

decision-mapping 不负责把需求写漂亮。它负责把未知数解决到“可以写 PRD”的程度。等关键问题被回答，下一阶段才进入 to-prd、domain-modeling 或 grill-with-docs。

回忆练习

判断应该直接进入下一阶段，还是先做 decision map。

1.“我们要把登录页按钮颜色从蓝色改成绿色。”

通常不需要 decision map。需求很小、未知数很少，可以直接实现或走普通 review。

2.“我们想把计费系统重做，但还不确定订阅、发票、试用期和迁移策略。”

适合 decision map。未知数多，而且彼此有依赖，需要分成 Discuss、Research、Prototype 票据逐个推进。

3.“我们知道要做一个 onboarding wizard，但不确定步骤是否太多，用户会不会卡住。”

可能先建 decision map，其中某些票据用 Prototype。这里的关键问题是交互体验，需要试出来。
