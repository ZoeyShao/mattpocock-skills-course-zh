完整工程 Flow 复盘

第 0011 课 · 总复盘 · 18 分钟

你现在可以把这套 skills 看成一个工程系统，而不是一堆命令。

主 flow：idea → ship

  1. setup-matt-pocock-skills：准备 repo 约定。
  2. decision-mapping / grilling / prototype：澄清未知数。
  3. to-prd + domain-modeling：写需求并稳定语言。
  4. codebase-design / improve-codebase-architecture：定接口、seam 和 deepening 方向。
  5. to-issues：拆成纵向切片。
  6. implement / tdd / diagnosing-bugs：实现和修复。
  7. review：按 spec 和 standards 两轴检查。
  8. handoff / writing / writing-great-skills：沉淀和交接。

两个 on-ramp

如果起点是“已有 issue 堆积”，从 triage 进入；如果起点是“代码库健康维护”，从 improve-codebase-architecture 进入。它们都会回到主 flow。

什么时候问 ask-matt

ask-matt 是路由器。当你不确定该从哪条流开始，或者忘了哪个 skill 对应哪个阶段，就问它。

从 30 个 skills 学到的写法

这些 skills 的共同目标不是“把内容写全”，而是让 agent 每次都更可预测地走同一套过程。下面是最值得迁移到你自己 skill 写作里的模式。

Takeaway 原 skill 线索 怎么借鉴
description 是触发条件，不是漂亮摘要。 to-issues frontmatter 写明：Use when user wants to convert a plan into issues... 写 description 时列出真实触发分支：用户说什么、任务长什么样、其他 skill 什么时候会依赖它。
用 leading word 压缩行为。 decision-mapping 用 fog of war；to-issues 用 tracer bullet；tdd 用 red-green-refactor。 不要反复解释同一行为。找一个稳定词，让它成为整段流程的抓手。
先定义耐久产物。 decision-mapping 把 map 称为 canonical artifact；prototype 说原型最后值得保留的是 answer。 skill 开头就回答：这次运行结束后，仓库里或 issue tracker 里应该留下什么。
用停止条件防止过早完成。 diagnosing-bugs 要求先有 red-capable feedback loop；tdd 明确：Never refactor while RED. 把“什么时候可以进入下一阶段”写成可检查条件，而不是写成模糊愿望。
反模式要直接写出来。 tdd 有 Anti-Pattern: Horizontal Slices；to-issues 强调每个 issue 不是横向层切片。 如果 agent 常犯某个错，不要只写正例；把错法命名并禁止。
把分支显式化。 decision-mapping 分 bootstrap / resume；prototype 分 logic / UI 两条 branch。 一个 skill 有多种运行方式时，先写“怎么选分支”，再写每条分支的动作。
写清楚要读哪些上下文。 setup-matt-pocock-skills 约定 issue tracker、triage labels、domain docs；grill-with-docs 会对照 CONTEXT.md 和 ADR。 不要假设 agent 会知道去哪找事实。把上下文入口写成 workflow 的一部分。
短 skill 也可以很强。 grilling、handoff、implement 都很短，但触发清楚、行为单一。 只有当它能防跑偏时才加长度。核心动作单一时，用短指令保留锋利度。
把 reference 下沉，别把主流程写肿。 writing-great-skills 用 GLOSSARY.md 承载术语；tdd 把 tests、mocking、deep modules、interface design 放到外部参考。 主 SKILL.md 保留每次都要执行的东西；只在某些分支需要的材料，放到被明确指向的参考文件。
主 skill 写总纲，延伸文档写深入路径。 codebase-design 先定义 module、interface、seam、depth 等共享语言，再用 DEEPENING.md 和 DESIGN-IT-TWICE.md 承接不同深入场景。 如果一个 skill 同时有“概念框架”和“专项操作手册”，不要全塞进主文件。主文件教 agent 识别问题和选择路径；专项文档负责把某条路径走深。

压缩成一句话：好 skill 不是知识库，而是带触发条件、产物、门槛和反模式的执行协议。

最终小测

“我们有一个想法，已经问清楚，也写了 PRD，现在要让多个 agent 并行实现。”下一步？

to-issues。把 PRD 拆成可独立验证的纵向切片，然后每个 issue 用新的实现会话处理。
