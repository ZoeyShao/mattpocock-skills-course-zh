术语解释

这份表只解释课程里反复出现、容易混淆的词。它不是翻译词典，而是帮助你判断这些 skills 在工程流程里到底指什么。

Skill 与工作流

术语意思在课程里怎么用
skill一个可被 agent 调用的工作流说明，通常由 SKILL.md 和少量参考文件组成。不要把 skill 当命令清单；它更像“什么时候用、怎么推进、产出什么”的操作规程。
SKILL.mdskill 的主说明文件。阅读时先看 frontmatter 里的 description，再看流程、约束和产物。
frontmatterMarkdown 文件顶部的 YAML 元数据块。用来声明 skill 名称、触发条件和简短用途。
agent执行任务的 AI 工作单元。课程里默认 agent 会读文档、改代码、跑检查，但需要明确上下文和边界。
handoff交接文档或交接动作。当上下文太长、要换会话或换 agent 时，用它保留当前状态和下一步。

仓库与代码

术语意思在课程里怎么用
reporepository，Git 仓库。它是代码存放和版本管理的单位。问“这个东西在哪个 repo 里”是在问存放位置。
codebase代码库，指一套项目代码作为工程对象的整体。它可能等于一个 repo，也可能只是 monorepo 的一部分，或跨多个 repo。
monorepo一个 Git 仓库里放多个 app、package 或服务。这时 repo 和 codebase 不一定一一对应。
single-context repo单上下文 repo，整个仓库基本共享一套领域语言和架构决策。agent 读根目录的 CONTEXT.md 和 docs/adr/ 通常就够了。
multi-context repo多上下文 repo，一个仓库里有多个相对独立的业务或技术上下文。不同上下文可能有不同术语、规则和 ADR。

需求、任务与状态

术语意思在课程里怎么用
PRDProduct Requirements Document，产品需求文档。把模糊想法整理成可实现、可验收的需求。
issue工作票据或任务卡，不只是 bug。可以表示 bug、功能、重构、QA 反馈，或从 PRD 拆出的小任务。
issue tracker管理 issue 的系统，例如 GitHub Issues、GitLab Issues、Linear 或本地 markdown。to-prd、to-issues、triage 都需要知道它在哪里。
triage对进入 issue tracker 的请求做分类、补信息、定状态。目标不是立刻实现，而是判断它能否被 agent 或人类接手。
triage labelissue tracker 里的状态或类型标签。例如 needs-info、ready-for-agent、wontfix。
QA feedback试玩、测试或验收时发现的问题反馈。它可以先变成 issue，再进入 triage。
tracer bullet issue纵向切片 issue，穿过必要层次形成一条窄但完整的可验证路径。反例是“一个 issue 只建数据库、另一个只写 API、另一个只写 UI”的横向切片。

领域语言与架构

术语意思在课程里怎么用
domain领域，项目真正关心的问题空间。例如 billing、ordering、invitation，而不是单纯的文件夹名。
domain language领域语言，团队对领域概念的稳定命名。它回答“这些东西叫什么、边界在哪里”。
DDDDomain-Driven Design，领域驱动设计。课程里主要借用它的术语思想，不要求完整实践 DDD 方法论。
ADRArchitecture Decision Record，架构决策记录。它回答“为什么我们这样设计”，适合记录有明显取舍的技术决定。
seam代码里的接缝，系统可以被替换、隔离或测试的边界。判断 seam 时要问它是否真的有多个 adapter，还是过早抽象。
adapter适配器，把一个外部系统或具体实现包成内部可用的接口。多个 adapter 常常说明某个 seam 是真实存在的。
interface模块暴露给调用者的入口形状。好 interface 让调用者少知道内部细节。
deep module深模块，小接口背后承载较多复杂行为。codebase-design 和 improve-codebase-architecture 常用这个判断模块质量。

实现与质量

术语意思在课程里怎么用
TDDTest-Driven Development，测试驱动开发。按红灯、绿灯、重构循环推进一个行为。
red-green-refactor红灯、绿灯、重构。先写失败测试，再写最少实现让测试通过，然后整理代码。
feedback loop反馈循环，用一个可重复运行的检查告诉你改动是否有效。诊断 bug 时先建立 loop，再提出假设。
Standardsreview 的一条轴：代码是否符合仓库规范。看风格、架构约定、测试习惯、安全边界。
Specreview 的另一条轴：代码是否符合 issue 或 PRD 要求。防止代码写得漂亮但做错事。
pre-commit提交前自动运行的检查钩子。常用于格式化、类型检查、lint、测试。

探索与决策

术语意思在课程里怎么用
decision map把模糊想法拆成一组有依赖关系的问题票据。用于还不能直接写 PRD 的早期探索。
Discussdecision map 里的问题类型，靠讨论和取舍解决。常搭配 grilling 或 domain-modeling。
Researchdecision map 里的问题类型，靠读资料解决。例如第三方 API、竞品、规范或本地文档。
Prototypedecision map 里的问题类型，靠试出来解决。适合 UI 手感、状态机、交互流程或业务规则可行性。
AFK agent可以在人不盯着的情况下接手工作的 agent。ready-for-agent 的意思就是信息足够，agent 可以独立推进。
