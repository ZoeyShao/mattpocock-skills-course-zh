Deepening

How to deepen a cluster of shallow modules safely, given its dependencies. Assumes the vocabulary in SKILL.md — module, interface, seam, adapter.

Dependency categories

When assessing a candidate for deepening, classify its dependencies. The category determines how the deepened module is tested across its seam.

1. In-process

Pure computation, in-memory state, no I/O. Always deepenable — merge the modules and test through the new interface directly. No adapter needed.

我的批注：这类依赖只发生在当前进程内：纯函数、内存对象、数组、Map 等，没有数据库、文件、网络、邮件或第三方 API。因为没有外部世界要替换，可以直接把零散浅模块合并到一个更深的接口后面，并通过这个新接口测试，不需要 adapter 或 mock。

2. Local-substitutable

Dependencies that have local test stand-ins (PGLite for Postgres, in-memory filesystem). Deepenable if the stand-in exists. The deepened module is tested with the stand-in running in the test suite. The seam is internal; no port at the module's external interface.

我的批注：这类依赖确实有 I/O，但测试里能用本地替身跑起来，例如用 PGLite 代替 Postgres、用内存文件系统代替真实文件系统。它可以 deepening，但这个替身通常只是模块内部测试策略；不要为了测试方便，把 databaseAdapter 之类的参数暴露成业务模块的外部接口。

3. Remote but owned (Ports & Adapters)

Your own services across a network boundary (microservices, internal APIs). Define a port (interface) at the seam. The deep module owns the logic; the transport is injected as an adapter. Tests use an in-memory adapter. Production uses an HTTP/gRPC/queue adapter.

我的批注：这类依赖是“远程，但归你们控制”的服务，例如内部微服务、内部 API、队列或 gRPC 服务。核心规则仍然应该归 deep module 管；跨网络的 HTTP/gRPC/queue 只是 transport adapter。测试时换成 in-memory adapter，生产时换成真实网络 adapter，这样业务逻辑不会散落在 client、controller 或 handler 里。

Recommendation shape: "Define a port at the seam, implement an HTTP adapter for production and an in-memory adapter for testing, so the logic sits in one deep module even though it's deployed across a network."

4. True external (Mock)

Third-party services (Stripe, Twilio, etc.) you don't control. The deepened module takes the external dependency as an injected port; tests provide a mock adapter.

Seam discipline

  • One adapter means a hypothetical seam. Two adapters means a real one. Don't introduce a port unless at least two adapters are justified (typically production + test). A single-adapter seam is just indirection.
  • Internal seams vs external seams. A deep module can have internal seams (private to its implementation, used by its own tests) as well as the external seam at its interface. Don't expose internal seams through the interface just because tests use them.

Testing strategy: replace, don't layer

  • Old unit tests on shallow modules become waste once tests at the deepened module's interface exist — delete them.
  • Write new tests at the deepened module's interface. The interface is the test surface.
  • Tests assert on observable outcomes through the interface, not internal state.
  • Tests should survive internal refactors — they describe behaviour, not implementation. If a test has to change when the implementation changes, it's testing past the interface.

    我的批注：测试应该验证模块对外表现出来的行为，而不是验证模块内部怎么写。测试越过公开接口，钻进内部实现细节里；这样一重构，行为没变，测试也会坏，测试就变成了维护负担。
